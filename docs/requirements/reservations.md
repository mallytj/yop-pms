# Reservation API RTM

> Foundation requirements for the reservation domain. Edge cases live in section 8 — every edge case maps to a test.

## 1. Core CRUD

| ID              | Requirement Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| --------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| R-RES-CRUD-001  | Create a reservation with property_id, primary guest (or inline guest payload — see R-RES-CRUD-015), one or more reservation items (room type, rate plan, stay period per item), and optional group/travel agent                                                                                                                                                                                                                                                                                                 |
| R-RES-CRUD-002  | Read a single reservation by ID (including all reservation items, guest details, and folio summary)                                                                                                                                                                                                                                                                                                                                                                                                              |
| R-RES-CRUD-003  | Read a list of reservations filtered by the parameters defined in section 9 (cursor pagination)                                                                                                                                                                                                                                                                                                                                                                                                                  |
| R-RES-CRUD-004a | `PATCH /reservations/{id}` updates reservation-level metadata only: notes, group_id, travel_agent_id, primary_guest_id. Requires `If-Match: <version>` header (R-RES-VALID-012)                                                                                                                                                                                                                                                                                                                                  |
| R-RES-CRUD-004b | `PATCH /reservations/{id}/items/{item_id}` updates item-level fields: stay_period, booked_room_type_id, rate_plan_id, guest_id, adults_count, children_count. Triggers availability re-check, ledger row move, booked_daily_rates recompute in single transaction. Requires `If-Match: <item_version>`                                                                                                                                                                                                           |
| R-RES-CRUD-005  | `POST /reservations/{id}/cancel` body `{ reason_code, reason_notes, fee_pence (nullable), waive_fee (bool), refund_action: none\|refund_deposit\|hold_for_review }`. Side-effects in single tx: reservation.status=`cancelled`, all non-terminal items → `cancelled`, future-dated ledger rows deleted, fee posted as folio_transaction on Folio A (unless `waive_fee`), `NOTIFY reservation_changes`, cancellation email enqueued via outbox. `reservations:waive_fee` permission required for `waive_fee=true` |
| R-RES-CRUD-006  | Reactivate a cancelled reservation (set `status` back to `confirmed`, clear `deleted_at`). Reactivation re-checks availability and re-pins ledger rows; returns 409 if dates are no longer available                                                                                                                                                                                                                                                                                                             |
| R-RES-CRUD-007  | Reservation code format `RES-XXXXXX`, sequential per property. Backed by `operations.reservation_sequences (property_id PK, next_value)` table; incremented in tx via `SELECT … FOR UPDATE`. Schema migration M1                                                                                                                                                                                                                                                                                                 |
| R-RES-CRUD-008  | Reservation creation generates an audit log entry via the transactional outbox worker                                                                                                                                                                                                                                                                                                                                                                                                                            |
| R-RES-CRUD-009  | Reservation creation produces ≥1 reservation_item and one `pricing.booked_daily_rates` row per item per night in stay_period (atomic with reservation insert). Source `walkin` and `internal` may pre-fill `assigned_room_id`; `website` defers explicit assignment but ledger auto-pins (R-RES-AVAIL-009)                                                                                                                                                                                                       |
| R-RES-CRUD-010  | Reservation creation must create at least one folio (Folio A — stay charges). Folios B and C are created lazily by staff. No folio routing rules at this stage                                                                                                                                                                                                                                                                                                                                                   |
| R-RES-CRUD-011  | Each reservation item must be associated with a room inventory ledger entry to ensure no overbooking                                                                                                                                                                                                                                                                                                                                                                                                             |
| R-RES-CRUD-012  | All entities must be isolated by authenticated `property_id` to prevent cross-property data access (RLS via `app.current_property_id`)                                                                                                                                                                                                                                                                                                                                                                           |
| R-RES-CRUD-013  | Reservation lifecycle: `hold` is the default state on creation for sources `website` and `internal`. `walkin` flag bypasses hold and lands directly in `checked_in` (R-RES-CRUD-014). OTA inbound (section 10) lands in `confirmed`                                                                                                                                                                                                                                                                              |
| R-RES-CRUD-014  | `is_walkin: true` on POST requires `lower(stay_period) = today`, `assigned_room_id` set on every item, and creates reservation + items in `checked_in` state, ledger rows in `sold`. Source remains `internal`                                                                                                                                                                                                                                                                                                   |
| R-RES-CRUD-015  | Inline guest creation: POST accepts `primary_guest_id` (existing) OR `primary_guest` (full guest payload, created in same tx). Same option per item for additional guests. Email-based dedup: matching active guest is reused, returned with `was_existing: true`                                                                                                                                                                                                                                                |

## 2. Availability & Conflict Prevention

| ID              | Requirement Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| --------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| R-RES-AVAIL-001 | Before creating/updating a reservation, the system MUST check room availability for the requested stay period                                                                                                                                                                                                                                                                                                                                                                                                      |
| R-RES-AVAIL-002 | Availability check reads `inventory.room_inventory_ledger` (statuses `sold`, `on_hold`, `maintenance`, `decommissioned`) for the requested period. Maintenance and decommissioning written to ledger directly (M3); availability is single-source-of-truth via the ledger                                                                                                                                                                                                                                          |
| R-RES-AVAIL-003 | Per-room overlap prevented at DB level via existing EXCLUDE constraint on `reservation_items (assigned_room_id WITH =, stay_period WITH &&)` AND `inventory.room_inventory_ledger UNIQUE(room_id, calendar_date)`. Per-room-type overlap prevented at application level via aggregate count: `available_count = total_rooms_in_type - ledger_rows_blocking(date)` per date in stay_period. Hold creation auto-pins a specific room (R-RES-AVAIL-009) so per-type race resolves to per-room race, which DB prevents |
| R-RES-AVAIL-004 | Concurrent reservation attempts for the same room/date range MUST be serialised (see ADR-013 for locking strategy)                                                                                                                                                                                                                                                                                                                                                                                                 |
| R-RES-AVAIL-005 | Availability check must include buffer periods (e.g. housekeeping turnaround) per property setting `housekeeping_buffer_minutes` (M4)                                                                                                                                                                                                                                                                                                                                                                              |
| R-RES-AVAIL-006 | The system must report which specific dates conflict when a reservation is rejected for overlap                                                                                                                                                                                                                                                                                                                                                                                                                    |
| R-RES-AVAIL-007 | Availability response must include remaining inventory count per room type for the requested dates                                                                                                                                                                                                                                                                                                                                                                                                                 |
| R-RES-AVAIL-008 | Availability check must satisfy all length of stay, occupancy and further restrictions defined in the price grid for the requested room type and dates (consolidated with R-RES-RATE-005)                                                                                                                                                                                                                                                                                                                          |
| R-RES-AVAIL-009 | On `hold` reservation creation, system selects an available room of the requested type (deterministic policy: lowest room number) and writes ledger rows for each calendar_date in stay_period. `assigned_room_id` on item stays NULL until staff confirms; ledger row may be moved to a different room of the same type without changing `assigned_room_id`                                                                                                                                                       |
| R-RES-AVAIL-010 | Hold expiry handled by background worker (section 12). Per-source TTL configurable per property                                                                                                                                                                                                                                                                                                                                                                                                                    |
| R-RES-AVAIL-011 | On reservation cancellation or check-out before `upper(stay_period)`, ledger rows for unused future calendar_dates are deleted                                                                                                                                                                                                                                                                                                                                                                                     |
| R-RES-AVAIL-012 | Maintenance blocks write `inventory.room_inventory_ledger` rows with status `maintenance` for each calendar_date in `block_period`. UNIQUE(room_id, calendar_date) prevents reservation/maintenance overlap at DB level                                                                                                                                                                                                                                                                                            |

## 5. Validation & Constraints

| ID              | Requirement Description                                                                                                                                                                                                                                                                                                                                   |
| --------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| R-RES-VALID-001 | Each `reservation_item.stay_period` must be chronological (`lower < upper`), both bounds required. DB CHECK enforces. A multi-item reservation may have items with differing stay_periods                                                                                                                                                                 |
| R-RES-VALID-002 | `lower(stay_period)` must not be earlier than today (UTC, property-tz aware). Exception: caller has `reservations:retroactive_create` permission OR `is_walkin=true` (today only)                                                                                                                                                                         |
| R-RES-VALID-003 | Stay period must not exceed the configured maximum stay length (property-level setting)                                                                                                                                                                                                                                                                   |
| R-RES-VALID-004 | All reservation mutations require `Idempotency-Key` header (POST/PATCH) — see ADR-007                                                                                                                                                                                                                                                                     |
| R-RES-VALID-005 | Reusing an Idempotency-Key with a different request body must return 409 Conflict                                                                                                                                                                                                                                                                         |
| R-RES-VALID-006 | Reservation status transitions must follow valid state machine rules (see section 7)                                                                                                                                                                                                                                                                      |
| R-RES-VALID-007 | Cancelled reservations must not be modifiable except via reactivation (return 409 if status is cancelled and request is not reactivate)                                                                                                                                                                                                                   |
| R-RES-VALID-008 | Notes field must respect database constraint (max 2500 characters)                                                                                                                                                                                                                                                                                        |
| R-RES-VALID-009 | Reservation code must be unique per property (enforced at DB level)                                                                                                                                                                                                                                                                                       |
| R-RES-VALID-010 | Reservation must respect minimum/maximum occupancy constraints of the room type                                                                                                                                                                                                                                                                           |
| R-RES-VALID-011 | A walkin reservation must be created on the same day, a room must be immediately assigned (see R-RES-CRUD-014)                                                                                                                                                                                                                                            |
| R-RES-VALID-012 | All mutating endpoints (`PATCH /reservations/{id}`, `PATCH /reservations/{id}/items/{item_id}`, `POST .../cancel`, `POST .../reactivate`) require `If-Match: <version>` header. Version mismatch returns 412 Precondition Failed with `{ current_version, server_value }`. Successful mutation bumps `version` by 1. Items have own `version` column (M2) |

## 6. Integration Points

| ID              | Requirement Description                                                                                                                                                                                                                                                                |
| --------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| R-RES-INTEG-001 | Cache reservation queries (individual and list) via `platform/cache` with TTL and pattern-based invalidation                                                                                                                                                                           |
| R-RES-INTEG-002 | Invalidate reservation cache on create, update, cancel, and room assignment                                                                                                                                                                                                            |
| R-RES-INTEG-003 | Emit `NOTIFY reservation_changes` on PostgreSQL for reactive cache invalidation and future real-time updates                                                                                                                                                                           |
| R-RES-INTEG-004 | Enqueue notification emails (confirmation, cancellation) via the transactional outbox worker outside of transaction                                                                                                                                                                    |
| R-RES-INTEG-005 | Rate calculation must call `GetOrSet` on the price grid cache for performance                                                                                                                                                                                                          |
| R-RES-INTEG-006 | Reservation state changes must be traceable via OpenTelemetry spans                                                                                                                                                                                                                    |
| R-RES-INTEG-007 | Hold-expiry worker scans for `hold` reservations exceeding source-specific TTL (`website_hold_ttl_seconds`, `internal_hold_ttl_seconds` — property settings, M4). Action: cancel reservation, delete ledger rows, mark related checkout_session `expired`, NOTIFY                      |
| R-RES-INTEG-008 | Archival worker scans for terminal reservations (`checked_out`, `cancelled`) older than `reservation_archive_after_days` (M4, default 365). Sets `status='archived'` and soft-archives related items + folios. Excluded from default list queries; opt-in via `?include_archived=true` |

## 7. State Machines

Reservation and item have separate state machines. Item transitions drive reservation rollup (see ADR-015).

### 7.1 Reservation status (`operations.reservation_status`)

| State         | Allowed Transitions           | Notes                                                                             |
| ------------- | ----------------------------- | --------------------------------------------------------------------------------- |
| `hold`        | → `confirmed`, → `cancelled`  | Default on website/internal POST. Auto-cancel via R-RES-INTEG-007 on TTL expiry   |
| `confirmed`   | → `checked_in`, → `cancelled` | Default for OTA inbound. Item rollup may transition to `checked_in` automatically |
| `checked_in`  | → `checked_out`               | Set when first item transitions to `checked_in`                                   |
| `checked_out` | → `archived` (via worker)     | Set when all items reach terminal state (`checked_out`, `no_show`, `cancelled`)   |
| `cancelled`   | → `confirmed` (reactivation)  | Reactivation re-checks availability + re-pins ledger                              |
| `archived`    | _(terminal)_                  | Set by R-RES-INTEG-008                                                            |

### 7.2 Reservation item status (`operations.reservation_item_status`)

| State         | Allowed Transitions                        | Notes                                                                                    |
| ------------- | ------------------------------------------ | ---------------------------------------------------------------------------------------- |
| `booked`      | → `checked_in`, → `no_show`, → `cancelled` | Default on item insert                                                                   |
| `checked_in`  | → `checked_out`                            | Requires `assigned_room_id NOT NULL`                                                     |
| `checked_out` | → `archived`                               | Set on guest checkout                                                                    |
| `no_show`     | → `cancelled`, → `archived`                | Set manually (`reservations:mark_no_show`) or by no-show worker after configurable grace |
| `cancelled`   | → `archived`                               | Set by reservation cancel cascade or item-level cancel                                   |
| `archived`    | _(terminal)_                               | Set by R-RES-INTEG-008                                                                   |

### 7.3 Rollup rule (item → reservation)

- All items `cancelled` → reservation `cancelled`
- All items terminal AND ≥1 `checked_out` → reservation `checked_out`
- ≥1 item `checked_in` AND no items `booked` → reservation `checked_in`
- Otherwise reservation status unchanged

Captured in ADR-015.

## 8. Edge Cases

> Each edge case must map to a test. Expected Behaviour references the requirement that resolves it.

| ID             | Edge Case                                                                                          | Expected Behaviour                                                                                                            | Test Status |
| -------------- | -------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------- | ----------- |
| R-RES-EDGE-001 | Concurrent booking of the last available room of a type                                            | First commit wins via R-RES-AVAIL-003 (DB UNIQUE on ledger). Loser receives 409 with conflicting dates                        |             |
| R-RES-EDGE-002 | Date update on a checked in reservation                                                            | Requires `reservations:post_checkin_mutate`. Stay can be extended only; shortening governed by R-RES-EDGE-035                 |             |
| R-RES-EDGE-003 | Reactivating a reservation when original dates are now conflicted                                  | R-RES-CRUD-006 re-checks availability. 409 with conflict dates if unavailable                                                 |             |
| R-RES-EDGE-004 | Reservation spanning a rate plan boundary                                                          | Each night resolved against the rate plan in force on that calendar date (R-RES-RATE-001)                                     |             |
| R-RES-EDGE-005 | Cancelling one item in a multi-room reservation                                                    | Item-level cancel via PATCH item; reservation rollup unchanged unless all items terminal                                      |             |
| R-RES-EDGE-006 | Idempotency key retry with identical body                                                          | ADR-007: cached response returned                                                                                             |             |
| R-RES-EDGE-007 | Booking attempted against a room in a maintenance block                                            | R-RES-AVAIL-012: ledger UNIQUE rejects insert. 409 with maintenance block dates                                               |             |
| R-RES-EDGE-008 | Maintenance required during stay period                                                            | Maintenance block creation rejected if overlapping with `sold` ledger rows. Staff must reassign first                         |             |
| R-RES-EDGE-009 | Guest has open folio balance at point of cancellation                                              | Cancel proceeds; folio balance preserved. Refund flow per `refund_action` in R-RES-CRUD-005                                   |             |
| R-RES-EDGE-010 | Rate plan deactivated or deleted mid stay                                                          | `rate_plan_id` ON DELETE SET NULL. Existing booked_daily_rates preserved (snapshot)                                           |             |
| R-RES-EDGE-011 | Checked in room now exceeding max length of stay                                                   | Existing reservations grandfathered. Future LOS changes don't retroactively invalidate                                        |             |
| R-RES-EDGE-012 | OTA booking concurrently with manual booking                                                       | OTA inbound queued; same DB constraints (R-RES-AVAIL-003) apply. Loser dead-lettered after retries                            |             |
| R-RES-EDGE-013 | Stagnant lock on room preventing further booking                                                   | R-RES-INTEG-007 (hold expiry worker) sweeps stale holds                                                                       |             |
| R-RES-EDGE-014 | Availability check requested for a date range with no configured rates                             | Returns 200 with `available: false` per affected dates and `reason: no_rate_configured`                                       |             |

| R-RES-EDGE-017 | Two reservations with adjacent date ranges (check-in & check-out overlap)                          | TSTZRANGE upper bound exclusive; adjacent ranges do not overlap. Allowed unless housekeeping buffer applies (R-RES-AVAIL-005) |             |
| R-RES-EDGE-018 | Availability check spanning a daylight saving boundary                                             | All times stored TIMESTAMPTZ (UTC). DST handled by tz conversion at presentation layer                                        |             |
| R-RES-EDGE-019 | Availability check during a leap year                                                              | DATE arithmetic in Postgres handles correctly                                                                                 |             |
| R-RES-EDGE-020 | All rooms assigned, but a cancellation during availability check                                   | Eventually consistent; cursor-stable list. Final commit re-checks via R-RES-AVAIL-001                                         |             |

| R-RES-EDGE-022 | Derived rate plan that has been updated                                                            | New rate applies to new bookings only. Existing booked_daily_rates snapshot preserved                                         |             |

| R-RES-EDGE-025 | Derived rate plan resulting in negative nightly rate after adjustment                              | Clamped to 0 with warning logged. Override required for true comp                                                             |             |
| R-RES-EDGE-026 | Rate adjustment resulting in negative nightly rate                                                 | Same as EDGE-025                                                                                                              |             |
| R-RES-EDGE-027 | No rate for a day during stay period                                                               | Booking rejected with explicit missing-date list                                                                              |             |

| R-RES-EDGE-029 | Multi-rate reservation — rate plan is not consistent across stay                                   | Allowed via per-night `booked_daily_rates.rate_plan_id`                                                                       |             |
| R-RES-EDGE-030 | Primary guest record soft-deleted or redacted                                                      | Reservation read returns redacted guest payload. Mutations require new guest assignment                                       |             |
| R-RES-EDGE-031 | Simultaneous room assignment to different reservations                                             | DB EXCLUDE constraint on `assigned_room_id` rejects loser                                                                     |             |
| R-RES-EDGE-032 | Guest added to reservation item at maximum occupancy                                               | Rejected per R-RES-VALID-010                                                                                                  |             |
| R-RES-EDGE-033 | Room type deactivated after confirmation and before stay                                           | Existing reservation preserved. Deactivation prevents new bookings only                                                       |             |
| R-RES-EDGE-034 | Stay period extended where new check-out conflicts with another reservation simultaneously created | R-RES-AVAIL-003 + R-RES-VALID-012 (If-Match) — loser receives 409 or 412                                                      |             |

| R-RES-EDGE-038 | Reservation rate changed during stay                                                               | PATCH item rate_plan_id; future booked_daily_rates recomputed, past preserved                                                 |             |
| R-RES-EDGE-039 | Reservation rate changed before check-in                                                           | PATCH item rate_plan_id; full recompute                                                                                       |             |
| R-RES-EDGE-040 | Cancellation attempted on checked-out reservation                                                  | 409 Conflict — terminal state, not cancellable                                                                                |             |
| R-RES-EDGE-041 | Reactivation attempted on past reservation                                                         | 409 Conflict if `lower(stay_period) < today` (unless `reservations:retroactive_create`)                                       |             |
| R-RES-EDGE-042 | Reservation cancelled and reactivated multiple times                                               | Allowed; each cycle audited                                                                                                   |             |
| R-RES-EDGE-043 | Room lock acquired but server crashed before reservation is committed                              | R-RES-INTEG-007 (TTL worker) recovers stale holds                                                                             |             |
| R-RES-EDGE-044 | Two requests with the same Idempotency key arrive simultaneously (parallel retry)                  | ADR-007: atomic Redis SET NX; second request waits or 409s                                                                    |             |
| R-RES-EDGE-045 | Reservation update and cancellation submitted concurrently for the same reservation                | R-RES-VALID-012 (If-Match) — second mutation receives 412                                                                     |             |
| R-RES-EDGE-046 | Reservation being created and server crashes                                                       | Tx rolled back. Idempotency-Key replay completes the create                                                                   |             |
| R-RES-EDGE-047 | Availability check shows room is available, but no longer available at the time of booking         | Hold auto-pins ledger row immediately on POST (R-RES-AVAIL-009). Race resolves to per-room DB conflict                        |             |
| R-RES-EDGE-048 | Attempting to check in an item that has no room assigned                                           | Rejected per R-RES-GROOM-003 + state transition guard                                                                         |             |

| R-RES-EDGE-051 | Checked in reservation has `PATCH` attempted on dates                                              | Requires `reservations:post_checkin_mutate`. Past nights immutable                                                            |             |
| R-RES-EDGE-052 | Batch status updated attempted on a mix of items at various states. Some invalid transitions       | Per-item validation. Response 207 Multi-Status with per-item result                                                           |             |
| R-RES-EDGE-053 | Outbox worker fails to deliver confirmation email                                                  | Existing outbox retry with exponential backoff; dead-letter after N attempts                                                  |             |
| R-RES-EDGE-054 | Redis cache returns stale availability data during invalidation race                               | Cache TTL fallback; final commit re-checks at DB                                                                              |             |
| R-RES-EDGE-055 | OpenTelemetry span fails to initialise                                                             | Logged at warn; reservation operation proceeds                                                                                |             |
| R-RES-EDGE-056 | Reservation created successfully but `NOTIFY` fails                                                | Cache TTL fallback; consumers eventually consistent                                                                           |             |
| R-RES-EDGE-057 | Length of stay requirements updated whilst reservations exist over the new requirement             | Existing reservations grandfathered; only new bookings enforce updated LOS                                                    |             |
