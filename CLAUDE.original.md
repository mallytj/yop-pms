# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

**Yop PMS** is a Property Management System built as a monorepo with end-to-end type safety from the database to the frontend. The stack is Go (Chi router) + SvelteKit 5 (Runes) + PostgreSQL + Redis, with SQLC for type-safe database access and OpenAPI as the contract between backend and frontend.

## Common Commands

All commands run from the repo root via `make`.

### Development
```bash
make setup       # First-time setup: install Go tools, create .env, install npm deps
make docker-up   # Start PostgreSQL, Redis, Adminer, Redis Commander, and server
make db-up       # Start only PostgreSQL + Redis (if running Go locally via Air)
make dev         # Start Go (Air hot-reload) + SvelteKit (Vite) concurrently
```

### Code Generation
```bash
make gen                # Full API contract generation: Swagger → OpenAPI → TypeScript types
make swag               # Generate Swagger/OpenAPI specs from Go comments only
make sqlc               # Generate type-safe DB code from SQL queries
make gen-constraints    # Sync config/constraints.g.yml + web/src/lib/types/constraints.g.ts from live DB
```
**Always run `make gen` after modifying Swagger comments in Go code.** The pre-push hook enforces this automatically.
**Run `make gen-constraints` after adding or changing DB check constraints** to keep the generated YAML and TypeScript in sync.

### Testing
```bash
make test          # Run all tests (backend + frontend)
make test-backend  # go test -race ./...
```

### Quality Checks
```bash
make audit    # Full quality suite: go mod verify, go vet, tests, govulncheck, svelte-check
make lint     # golangci-lint (Go) + Prettier validation (frontend)
make format   # go fmt (Go) + Prettier (frontend)
```

### Database
```bash
make reset-db       # Full DB teardown and restart (docker-compose down -v + up)
make goose-circle   # Complete Goose migration reset
```

## Architecture

### API Contract Flow (Schema-First)
1. Swagger annotations in Go handlers → `make swag` → `/api/swagger.json`
2. Convert to OpenAPI 3.0 → `/api/openapi.json`
3. Generate TypeScript types → `/web/src/lib/types/api.d.ts`

The frontend **must not** define its own API types — they come from generation. If `api.d.ts` is stale, TypeScript compilation fails.

### Backend (`cmd/` + `internal/`)
- Entry point: `cmd/server/main.go` (config, DB pool, Redis, server init) + `cmd/server/api.go` (route registration)
- Domain logic lives in `internal/{booking,planner,pricing}/`
- Cross-cutting concerns in `internal/platform/` (config, db, middleware, logging, apierror, cache, constraints, events, helpers, json, otel)
- Data access in `internal/store/` — **all generated by SQLC, never edit manually**
  - Write SQL queries in `internal/store/queries/`, then run `make sqlc`

### Frontend (`web/`)
- SvelteKit 5 with **Runes** for reactivity (use `$state`, `$derived`, `$effect` — not legacy stores where possible)
- Path aliases: `$lib`, `$components`, `$helpers`, `$stores`, `$types`, `$actions`
- API types imported from `$types` (generated `api.d.ts`)

### Constraint Consistency System

DB check constraints are synced to two generated files — **never edit manually**:
- `config/constraints.g.yml` — machine-readable constraint metadata (used by backend validation)
- `web/src/lib/types/constraints.g.ts` — TypeScript types for frontend validation

Run `make gen-constraints` (requires live DB) to regenerate both. See `docs/guides/backend-constraints.md` and `docs/guides/frontend-constraints.md` for usage.

### Database Conventions (enforced in migrations)
- **Financials:** `INTEGER` only (smallest currency unit — no floats ever)
- **Timestamps:** `TIMESTAMPTZ` exclusively
- **Primary keys:** UUIDv7 (time-sortable)
- **Foreign keys:** `RESTRICT` on delete (preserve history)
- **Soft deletes:** `deleted_at TIMESTAMPTZ` — uniqueness indexes use `WHERE (deleted_at IS NULL)`
- **Multi-tenancy:** Every tenant-isolated table has `property_id`; Row-Level Security is enabled
- **Optimistic locking:** `version` column on high-concurrency tables
- **Text fields:** Check constraints on length; `CITEXT` for emails, usernames, codes
- Constraint naming: `{table}_{column}_{suffix}`

### Migrations
Managed by Goose in `/migrations/`. Migration filenames are zero-padded sequential numbers. Configure via `GOOSE_*` env vars in `.env`.

|---|---|---|
| PostgreSQL 18 | 5433 | Primary database |
| Redis 7 | 6379 | Cache / sessions |

Swagger UI: `http://localhost:8080/swagger/index.html`

## Key Decisions (see `/docs/adr/` for full context)
- **No ORM** — raw SQL via SQLC for type safety without abstraction overhead
- **Pure CSS** on frontend — no CSS framework
- **`scratch`-based Docker image** — ~30MB final image, non-root user
- **Redis** is for transient state only (sessions, token blacklists, rate limiting) — not a primary store
